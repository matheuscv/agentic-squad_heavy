Serviços de Infraestrutura — Detalhamento

8 integrações · 3 camadas lógicas
J
Jira Cloud
Trigger & State Machine

Âncora do fluxo. Cada transição de card dispara um webhook HTTP que o Render consome. O orquestrador escreve de volta via REST API v3, movendo cards e adicionando comentários com links para artefatos gerados.

REST API v3 Webhooks HMAC 13 estados Basic Auth
GH
GitHub
VCS · CI/CD

Repositório do código gerado e plataforma de CI. O GitHub App autentica via JWT, permitindo criar commits, abrir PRs e disparar workflows. Os resultados de cobertura do Actions voltam ao Render para análise do Agente QA.

GitHub App Actions CI JWT + Install Token PR Review
R
Render
Web Service · Hub Central

Serviço long-running Node.js hospedando o orquestrador, todos os agentes e os workers BullMQ. Sem timeout de serverless — garante execuções de até 15 min por agente DEV. Deploy automático via Git push.

Express BullMQ Workers Auto-deploy Graceful Shutdown
UR
UptimeRobot
Monitoramento de Disponibilidade

Monitora o endpoint /health do Render a cada 30 segundos. Dispara alertas imediatos (e-mail/Slack) em caso de downtime, garantindo SLA e visibilidade executiva em tempo real.

HTTP Monitor 30s interval Alertas Push /health endpoint
SB
Supabase
PostgreSQL · Storage

Banco relacional centraliza todo estado: histórias, execuções de agentes, artefatos gerados e métricas de custo de tokens. O Supabase Storage persiste PRDs e Planos como blobs acessíveis por link.

Drizzle ORM stories table agent_runs Migrations
UP
Upstash Redis
BullMQ · Fila de Jobs

Backend serverless do BullMQ. Cada evento Jira vira um job enfileirado com retry, prioridade e dead-letter queue. Concorrência de até 5 workers DEV simultâneos sem race conditions.

BullMQ Concurrency=5 DLQ Exponential Backoff
BS
Betterstack
Logs · Alertas · Observabilidade

Recebe logs estruturados em JSON contendo story_id, agent, phase, duration_ms e token_cost. Alertas configuráveis disparam quando custo por história excede threshold.

JSON Logs Token Cost Threshold Alerts Source Token
∆
Anthropic API
LLM · Geração de Conteúdo

Cérebro de cada agente. Opus 4 para Orquestrador, PO, LT e QA — raciocínio profundo. Sonnet 4 para os 5 agentes DEV — equilíbrio custo/performance. Cada chamada instrumentada para auditoria de custo.

claude-opus-4-7 claude-sonnet-4-6 Tool Use Token Metering

Fluxo de Dados — Ciclo de Vida de uma História

Ponta a ponta · Backlog → Concluído
01
Disparo via Jira

Membro da equipe move card de Backlog para A Refinar. Jira dispara webhook assinado com HMAC-SHA256 para /webhooks/jira do Render em <5s.

Jira → Render → Upstash Queue
02
Agente PO gera PRD

Orquestrador detecta transição Em Refinamento, invoca Agente PO. Claude Opus lê a história via Jira API e gera PRD.md com RFs numerados e critérios mensuráveis. Commit automático no GitHub.

Render/PO → Anthropic → GitHub
03
Gate Humano — PRD

Card para em Aguardando Aceite PRD. Sistema aguarda ação humana — não avança automaticamente. Jira Reconciliation Job verifica a cada 1-2min se card foi aprovado. Aprovação: card → PRD Aceito.

Gate Humano → Jira Status → Reconciliação
04
Agente LT gera Plano

Agente LT lê o PRD.md e o README do repositório. Claude Opus decompõe em tasks técnicas numeradas (TASK-01, 02 ...), identificando paralelas vs. sequenciais. PLANO_DE_EXECUCAO.md commitado.

Render/LT → Anthropic → GitHub
05
Agentes DEV implementam (×5)

Após aprovação do Plano, até 5 Agentes DEV rodam em paralelo via BullMQ (concurrency=5). Cada DEV recebe sua task, implementa código + testes unitários, e abre PR com título [TASK-XX].

Upstash ×5 → Anthropic → GitHub PR
06
CI/CD + Agente QA

GitHub Actions executa lint, build, testes e relatório de cobertura. QA analisa resultado: cobertura <85% dispara loop de correção DEV↔QA (máx. 3 iterações). Aprovado: card → Aguardando Aceite QA. Logs e custos em Betterstack + Supabase.

GitHub CI → QA Agent → Betterstack
