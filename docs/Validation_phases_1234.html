Validar o ciclo completo de desenvolvimento agêntico — do Backlog ao Concluído — com verificação adicional de todos os mecanismos de hardening e resiliência da Fase 4.

F4-B

Probe: rate limiting — 60 req/min por IP

O webhook está configurado para aceitar no máximo 60 requisições por minuto por IP. A 61ª requisição na mesma janela recebe HTTP 429.

Script de teste (PowerShell):

# Envia 62 requisições e verifica que a 61ª retorna 429
$url = "https://<sua-app>.onrender.com/webhooks/jira"
$body = '{"issue":{"key":"TEST-1","fields":{"summary":"probe","status":{"name":"A Refinar"}}},"changelog":{"items":[{"field":"status","fromString":"Backlog","toString":"A Refinar"}]}}'
$headers = @{ "Content-Type" = "application/json" }

for ($i = 1; $i -le 62; $i++) {
    $resp = Invoke-WebRequest -Uri $url -Method POST -Body $body -Headers $headers -SkipHttpErrorCheck
    if ($resp.StatusCode -eq 429) {
        Write-Host "Req $i → HTTP 429 (rate limit aplicado corretamente)" -ForegroundColor Green
        break
    }
}
  • As primeiras 60 requisições retornam HTTP 200 (ou outro código de lógica, não 429)
  • A requisição número 61 (ou anterior, se a janela já estava parcialmente preenchida) retorna HTTP 429
  • Render Logs: não deve aparecer log de processamento para as requisições bloqueadas
⚠️ Execute este probe em horário de baixo tráfego

O rate limiter é in-memory por processo — o contador reseta quando o serviço reinicia ou quando a janela de 1 minuto expira. Se o Render tiver reiniciado recentemente, a janela começa zerada.

F4-C

Probe: replay protection — mesmo webhook enviado duas vezes

O webhook usa Redis SET NX EX 300 com uma chave que combina issueKey + fromStatus + toStatus + janela-de-5-min para bloquear reenvios dentro de 5 minutos — proteção contra retries automáticos do Jira e ataques de replay.

Script de teste:

# Use um Jira Key real que ainda NÃO foi processado nesta janela de 5 min
$url  = "https://<sua-app>.onrender.com/webhooks/jira"
$body = '{"issue":{"key":"SCRUM-99","fields":{"summary":"replay test","status":{"name":"A Refinar"}}},"changelog":{"items":[{"field":"status","fromString":"Backlog","toString":"A Refinar"}]}}'
$headers = @{ "Content-Type" = "application/json" }

# 1ª requisição — deve ser processada normalmente
$r1 = Invoke-WebRequest -Uri $url -Method POST -Body $body -Headers $headers
Write-Host "1ª: HTTP $($r1.StatusCode) — $($r1.Content)"

# 2ª requisição — idêntica, dentro de 5 min — deve retornar ignored
$r2 = Invoke-WebRequest -Uri $url -Method POST -Body $body -Headers $headers
Write-Host "2ª: HTTP $($r2.StatusCode) — $($r2.Content)"
  • 1ª requisição: HTTP 200 com corpo indicando processamento normal
  • 2ª requisição: HTTP 200 com corpo { "ignored": true, "reason": "replay_detected" }
  • Render Logs: apenas um log de "processando transição" — o segundo não deve ser processado
  • Jira board: card movido apenas uma vez, sem duplicação de estados
Por que HTTP 200?Retornar 200 para replays evita que o Jira continue retentando — a resposta indica "recebemos, mas já processamos".

F4-E

Observação: retry com backoff exponencial em APIs externas

Todas as chamadas para Jira e GitHub usam retry automático (3 tentativas, backoff 1 s → 2 s → 4 s) para erros 408, 429, 5xx. Em condições normais esses retries não ocorrem — mas são visíveis nos logs quando há instabilidade na API.

ClienteStatus retryáveisStatus não-retryáveis
Jira (jiraFetch) 408, 429, 500, 502, 503, 504 4xx exceto 408 e 429
GitHub (githubFetch) 429, 500, 502, 503, 504 401, 403 (falhas de autenticação)

Sinal de retry nos logs:

# Filtrar por "tentativa" nos logs do Render para ver retries em ação:
"tentativa 2 de 3 — aguardando 1000ms"
"tentativa 3 de 3 — aguardando 2000ms"

F4-F

Probe: Dead-Letter Queue (DLQ) e alerta no Jira

Quando um job falha nas 3 tentativas configuradas no BullMQ, ele é: (1) enviado para a fila agent-dlq, (2) um comentário de alerta é postado no Jira, e (3) o /health/detailed passa a retornar "status": "attention".

⚠️ Este é um teste destrutivo — opcional e controlado

Para simular uma falha de DLQ em produção você precisaria de uma história real que falhe. Em ambiente de homologação, valide pela inspeção do /health/detailed após um job que já tenha falhado naturalmente, ou pelo histórico de comentários no Jira de stories anteriores.

Como verificar o comportamento de DLQ após uma falha real:

  1. Acesse GET /health/detailed — se agent-dlq.waiting > 0, o status será "attention"
  2. No Jira, abra a story que falhou → aba Comments → procure o comentário do bot com o emoji ⚠️
  3. O comentário deve conter: nome do agente que falhou, número de tentativas, Job ID e Run ID
O quê verificarOndeEsperado
/health/detailed quando DLQ tem jobs Navegador ou curl "status": "attention" (HTTP 200, não 503)
Comentário de falha no Jira Card → aba Comments Comentário com ⚠️, nome do agente, tentativas, Job ID e Run ID
Job na fila DLQ Render Logs → filtrar "dead-letter" Log "job DLQ adicionado" com originalQueue
DLQ volta a zero após investigação /health/detailed depois de limpar "status": "ok" retorna após DLQ ser drenada
F4-G

Probe: startup recovery — sobrevivência a restart

Quando o serviço reinicia (deploy ou crash), todos os agentRuns com status = 'running' são automaticamente recuperados: resetados para 'pending' e re-enfileirados com o mesmo jobId (BullMQ deduplica se já estiver na fila).

Como testar:

  1. Inicie uma história e aguarde ela chegar ao estado Em Desenvolvimento (Agente DEV ativo — agentRun.status = 'running' no banco)
  2. No painel do Render: clique em Manual Deploy → Deploy latest commit (isso reinicia o serviço, simulando um crash)
  3. Observe os logs após o reinício
Render reinicia o processo └─ index.ts ou index-workers.ts inicializa └─ recoverInterruptedRuns({ po, lt, dev, qa }) ├─ consulta: agentRuns WHERE status = 'running' JOIN stories ├─ para cada run: UPDATE status = 'pending', startedAt = NULL ├─ re-enfileira: queue.add(jobName, jobData, { jobId }) └─ log: "runs interrompidos detectados — iniciando recovery de startup"

Logs a observar após o restart:

# No Render → Logs, nos primeiros segundos após o deploy:
"runs interrompidos detectados — iniciando recovery de startup"  count: N
"run reenfileirado com sucesso"  runId, agentType, jiraKey, jobId
"recovery de startup concluído"  recovered: N, total: N
  • Log "runs interrompidos detectados" com count > 0 logo após o restart
  • Log "run reenfileirado com sucesso" para cada run recuperado
  • Agente retoma a execução do job (aguarde alguns minutos — o job re-processa normalmente)
  • Jira board: card continua no estado correto após a recuperação (não regride)
  • Se reiniciar sem nenhum job ativo: log "nenhum run interrompido encontrado — startup limpo"
Por que isso é importante

O Render free tier reinicia automaticamente os workers periodicamente. Sem startup recovery, qualquer job em execução no momento do restart seria perdido silenciosamente. Com o recovery, o job é detectado pelo JOIN com a tabela stories (que preserva o jiraSummary) e re-processado com os dados corretos.

F4-H

Observação: graceful shutdown — SIGTERM via deploy normal

Todo deploy no Render envia SIGTERM ao processo antes de substituí-lo. O serviço para de aceitar novas conexões HTTP e drena os jobs BullMQ ativos (máx 30 segundos) antes de encerrar. Se o drain demorar mais de 30 s, o processo sai com código 1 (log de warning).

Logs a observar durante um deploy com agente ativo:

# Sequência esperada no Render → Logs durante deploy:
"sinal recebido — iniciando graceful shutdown"         signal: SIGTERM
"aguardando jobs ativos concluírem..."
"workers drenados"
"encerramento concluído"

# Se timeout de 30s for excedido (job muito longo):
"timeout de graceful shutdown — saindo forçado"         timeoutMs: 30000
  • Deploy sem agente ativo: sequência limpa de "graceful shutdown" → "encerramento concluído"
  • Deploy com agente ativo: sequência inclui "aguardando jobs ativos" → "workers drenados"
  • Ausência de "saindo forçado" em deploys normais (timeout só ocorre em jobs anormalmente longos)
