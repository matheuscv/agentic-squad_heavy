Os 11 Steps do Wizard — Guia Detalhado

1
📋 Jira
Configuração do Jira
Conecta a squad ao seu board de histórias e registra as credenciais de acesso
O que fazer

Informe a URL do seu workspace Jira, o e-mail da conta com acesso de administrador, o API Token (gerado em id.atlassian.com) e a chave do projeto (ex.: SCRUM). O webhook secret pode ser gerado automaticamente — basta pressionar Enter.

Para que serve

O wizard valida se as credenciais permitem acessar o projeto informado. Essas credenciais são usadas pelo servidor para mover cards automaticamente entre colunas e adicionar comentários com links para PRDs e Planos gerados pela IA.

JIRA_BASE_URLobrigatóriahttps://org.atlassian.net
JIRA_EMAILobrigatóriae-mail da conta Atlassian
JIRA_API_TOKENobrigatóriaid.atlassian.com → Security
JIRA_PROJECT_KEYobrigatóriaex.: SCRUM, MYAPP
JIRA_WEBHOOK_SECRETauto-geradoEnter = gera automaticamente
🔑

Para gerar o API Token: id.atlassian.com → Security → API tokens → Create API token. O token é global para toda a sua conta Atlassian — não é por projeto.

2
🐙 GitHub
Configuração do GitHub App
Autentica a squad para criar commits, branches e Pull Requests no seu repositório
O que fazer

Informe o dono e nome do repositório alvo. Forneça o App ID e o Installation ID do GitHub App (encontrados em github.com/settings/apps → seu app). Para a Private Key, informe o caminho do arquivo .pem ou cole o conteúdo inline.

Para que serve

O GitHub App é a identidade da squad no seu repositório. Com ele, os agentes DEV criam branches agent/task-XX, fazem commits atômicos e abrem Pull Requests com o código gerado. Toda a rastreabilidade fica no histórico do Git.

GITHUB_OWNERobrigatóriausuário ou organização
GITHUB_REPOobrigatórianome exato do repositório
GITHUB_APP_IDobrigatórianúmero na página do App
GITHUB_APP_INSTALLATION_IDobrigatóriaID da instalação no repo
GITHUB_APP_PRIVATE_KEYobrigatóriacaminho .pem ou inline \n
GITHUB_DEFAULT_BRANCHdefault: mainbranch principal do repo
⚠️

O Installation ID é diferente do App ID. Acesse github.com/settings/installations, clique no App instalado — o número na URL é o Installation ID.

3
🔍 Stack
Detecção Automática de Stack
Identifica a linguagem e configura os comandos de teste e cobertura do seu projeto
O que fazer

O wizard lê automaticamente os manifests do seu repositório (package.json, pom.xml, go.mod, etc.) e sugere os comandos de teste. Confirme ou ajuste conforme necessário. Na maioria dos casos, apenas pressione Enter para aceitar.

Para que serve

O Agente QA usa esses comandos para rodar os testes após cada ciclo de desenvolvimento. A cobertura medida é comparada ao threshold — abaixo disso, o loop DEV↔QA é ativado para corrigir automaticamente.

CI_TEST_COMMANDauto-detectadoex.: npm test, pytest
CI_COVERAGE_COMMANDauto-detectadoex.: npm run test:coverage
CI_COVERAGE_THRESHOLDdefault: 80%mínimo de cobertura exigido
🌐

Linguagens suportadas: Node.js (npm/yarn/pnpm/bun · vitest/jest/mocha), Python (pip/poetry/pdm · pytest), Java (maven/gradle), Go, Rust, Ruby, PHP.

4
🗄️ Supabase
Banco de Dados — Supabase / PostgreSQL
Persiste o estado de todas as histórias, execuções de agentes e métricas de custo
O que fazer

Acesse Supabase → Project Settings → Database → Connection string → URI e copie a URL. Cole no campo DATABASE_URL. O wizard valida a conexão com SELECT 1 e pergunta se deve executar as migrations automaticamente.

Para que serve

O banco é o estado central da squad. Todas as histórias em processamento, os artefatos gerados (PRD, Plano), os logs de execução por agente e os custos de tokens são persistidos aqui via Drizzle ORM.

DATABASE_URLobrigatóriapostgresql://postgres:...
SUPABASE_URLopcionalhttps://xxx.supabase.co
SUPABASE_ANON_KEYopcionalchave pública anon
SUPABASE_SERVICE_ROLE_KEYopcionalchave de service role
⚡

Aceite a execução das migrations quando o wizard perguntar — elas criam as tabelas stories, agent_runs e a estrutura do Drizzle. Sem as migrations o servidor não inicializa.

5
⚡ Redis
Fila de Jobs — Upstash Redis
Backend do BullMQ para enfileirar e processar tarefas dos agentes com retry e DLQ
O que fazer

Acesse Upstash Console → seu database → Details e copie a Connection String no formato rediss:// (com TLS). Cole no campo REDIS_URL. O wizard faz um PING para confirmar a conexão.

Para que serve

O Redis é o backbone do BullMQ. Cada evento Jira vira um job enfileirado com prioridade, retry exponencial e dead-letter queue (DLQ). Permite até 5 agentes DEV rodando em paralelo sem race conditions.

REDIS_URLobrigatóriarediss://default:...@host:port
⚠️

Use sempre a URL com prefixo rediss:// (dois s) para conexão TLS — o Upstash exige TLS em produção. URLs redis:// (sem TLS) funcionam apenas em ambientes locais.

6
🤖 Anthropic
Motor de IA — Anthropic API
A chave que dá inteligência a todos os agentes: PO, LT, DEV e QA
O que fazer

Acesse console.anthropic.com → Settings → API Keys e gere uma chave. Cole no campo ANTHROPIC_API_KEY. Em seguida, configure o modelo para cada agente ou pressione Enter para aceitar os defaults recomendados.

Para que serve

Cada agente usa o Claude para raciocinar sobre a história e gerar artefatos. Opus 4 é usado para PO, LT, QA e Orchestrator (raciocínio profundo). Sonnet 4 é usado para os 5 DEVs (menor custo, ótima geração de código).

ANTHROPIC_API_KEYobrigatóriask-ant-...
MODEL_ORCHESTRATORdefault: claude-opus-4-7aceita: opus / sonnet / haiku
MODEL_POdefault: claude-opus-4-7gera PRD.md
MODEL_LTdefault: claude-opus-4-7gera Plano.md
MODEL_QAdefault: claude-opus-4-7valida cobertura
MODEL_DEVdefault: claude-sonnet-4-6gera código (~$0.50/história)
💡

O wizard faz uma chamada de validação de ~$0.00003 (1 token) apenas para confirmar que a chave é válida. Você pode digitar atalhos: opus, sonnet ou haiku em vez do nome completo.

7
📊 Betterstack
Observabilidade — Betterstack + Thresholds
Logs JSON estruturados com contexto de agente, custo e story_id para rastreabilidade completa
O que fazer

Acesse betterstack.com → Telemetry → Sources → New Source, copie o Source Token e cole no wizard. Se deixado em branco, os logs ficam apenas no stdout do Render. Configure também os thresholds de concorrência e custo.

Para que serve

Com Betterstack, cada log carrega story_id, agent, phase, duration_ms e token_cost. Você pode criar alertas para histórias acima do custo esperado e dashboards de performance por agente.

BETTERSTACK_SOURCE_TOKENopcionaltelemetry token
AGENTS_DEV_CONCURRENCYdefault: 5DEVs paralelos máximos
COST_ALERT_THRESHOLD_USDdefault: 1.00alerta de custo por história
8
🚀 Render
Render — Configuração Completa do Serviço
Autentica na API do Render, seleciona o serviço e envia todas as variáveis automaticamente
O que fazer (3 sub-etapas)

8a — Informe a API Key do Render (Account Settings → API Keys). O wizard lista seus Web Services e você escolhe pelo número.

8b — Revisão das 26 variáveis — pré-preenchidas dos steps anteriores. Valores sensíveis são mascarados.

8c — O wizard pergunta se envia as variáveis via API. Se sim, o Render faz um novo deploy automaticamente.

Para que serve

Elimina o trabalho manual de copiar 26 variáveis no painel do Render. O wizard faz merge inteligente — preserva variáveis que você tenha configurado manualmente e sobrescreve apenas as do wizard. Após o envio, o Render inicia um deploy com a configuração completa.

RENDER_API_KEYwizard onlynão entra no .env.init
SERVICE_URLobrigatóriahttps://xxx.onrender.com
NODE_ENVdefault: production
PORTdefault: 3000
+ 26 vars revisadas e enviadas ao Rendervia APImerge inteligente
📁

O arquivo .env.render é sempre gravado como fallback — caso prefira configurar manualmente via render env set --service <id> < .env.render. A RENDER_API_KEY não é salva nos arquivos de saída.

9
⚙️ CI
Instalação do GitHub Actions Workflow
Instala o pipeline de CI nos branches dos agentes DEV do seu repositório alvo
O que fazer

Confirme a instalação quando o wizard perguntar. Ele fará um commit automático do arquivo .github/workflows/agent-dev.yml no seu repositório. Nenhuma interação adicional é necessária.

Para que serve

O workflow é ativado quando o Agente DEV faz push em branches agent/task-*. Ele roda lint, typecheck, build, testes e grava o relatório de cobertura em .qa-coverage.json — que o Agente QA lê para decidir se o PR está aprovado.

🔒

Após o wizard, configure Branch Protection no GitHub: Settings → Branches → Add rule → branch: main → Require status checks: "Lint · Typecheck · Build · Test · Coverage". Isso garante que o QA não pode aprovar um PR com CI quebrado.

10
🔗 Webhook
Registro do Webhook no Jira
Liga o Jira ao servidor Render — toda transição de card dispara a squad automaticamente
O que fazer

Confirme quando o wizard perguntar. Ele registra automaticamente o webhook via API do Jira com JQL project = "SEUKEY" e evento jira:issue_updated. A URL gerada inclui o secret para validação HMAC.

Para que serve

É a "campainha" da squad. Quando você mover um card no Jira, o webhook dispara um POST para /webhooks/jira do Render em menos de 5 segundos, iniciando o pipeline de agentes para aquela história.

⚠️

Webhooks dinâmicos do Jira expiram em 30 dias. Execute agentic-squad init novamente antes do vencimento para renovar. O wizard detecta os valores existentes no .env e preenche tudo automaticamente — leva menos de 1 minuto.

11
✅ Smoke Test
Smoke Test — Validação Ponta a Ponta
Cria uma história real no Jira, dispara o webhook e confirma que o pipeline responde corretamente
O que fazer

Confirme quando o wizard perguntar. Ele cria uma issue de teste com o título "[SMOKE TEST] agentic-squad init", move para o status trigger, aguarda a resposta do webhook e valida que retorna {"queued":true}. A issue é deletada automaticamente ao final.

Para que serve

Confirma que toda a cadeia está funcionando: Jira → Webhook → Render → BullMQ → fila de jobs. É a prova final de que a squad está pronta para receber histórias reais. Se falhar, o wizard indica exatamente qual parte da cadeia está quebrada.

✅

Resposta esperada: {"queued":true}. Se o serviço no Render ainda estiver em deploy após o Step 8, o smoke test será pulado automaticamente — execute o wizard novamente após o deploy estabilizar.

Próximos Passos — Após o Wizard Concluir

01
🔒 Branch Protection no GitHub
Acesse Settings → Branches → Add rule → main e ative Require status checks: Lint · Typecheck · Build · Test · Coverage. Impede que o QA aprove um PR com CI quebrado.
02
📊 UptimeRobot — Monitoramento
Configure manualmente: uptimerobot.com → Add Monitor → HTTP → URL: sua-url.onrender.com/health → Interval: 5 min. Você recebe alertas imediatos em caso de downtime.
03
🚀 Primeiro Deploy — Verificar Logs
Acesse o painel do Render e aguarde o deploy concluir. Verifique os logs iniciais — o servidor faz startup recovery automaticamente e conecta banco, Redis e workers.
04
📋 Primeira História Real no Jira
Crie uma história simples no seu board e mova de Backlog para A Refinar. Em menos de 30 segundos o Agente PO deve iniciar e gerar o PRD.md no GitHub.
05
⚑ Gates Humanos — Entender o Fluxo
A squad para em 3 momentos aguardando você: aceite do PRD, aprovação do Plano e merge do PR. O merge final é sempre uma decisão humana — nunca automatizada.
06
🔄 Renovar Webhook em 30 dias
Webhooks dinâmicos do Jira expiram em 30 dias. Execute agentic-squad init novamente — o wizard pré-preenche tudo do .env e renova apenas o webhook em < 1 min.
