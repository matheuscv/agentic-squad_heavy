Squad-in-a-Box
Guia de Configuração

Como reutilizar a Squad Agêntica em qualquer projeto — do zero ao fluxo completo em minutos.

Conceito

Como funciona a reusabilidade

O projeto funciona como um serviço centralizado — a squad é infraestrutura, não uma biblioteca embutida no projeto alvo. O projeto alvo recebe apenas dois artefatos e passa a evoluir suas features dentro do fluxo agêntico.

Artefatos

O que o projeto alvo recebe

⚙️
GitHub Actions Workflow
Arquivo .github/workflows/agent-dev.yml instalado diretamente no repo via API GitHub. Controla o gate de CI exigido pelos agentes DEV e QA.
🔗
Webhook Jira
Webhook registrado automaticamente em Jira Admin apontando para o serviço. Dispara quando uma issue muda para status de desenvolvimento.
⚠️
Atenção: Webhooks dinâmicos do Jira expiram em 30 dias. Re-execute agentic-squad init para renovar.
Pré-requisito

O serviço precisa estar rodando

A squad roda no Render (ou onde você fizer deploy). O SERVICE_URL aponta para essa instância. Se já há um deploy ativo, não é necessário subir nada novo — basta conectar o novo projeto.

🌐
Instância já deployada
Use o SERVICE_URL existente no Step 4 do wizard. A mesma instância serve múltiplos projetos.
🚀
Novo deploy
Clone este repo, configure as variáveis de ambiente e faça deploy no Render ou em qualquer plataforma Node.js.
Passo 1

Instalar o CLI (uma vez)

Instale o pacote globalmente para ter o comando agentic-squad disponível em qualquer diretório.

bash 
# instala o CLI globalmente
npm install -g agentic-squad-heavy

# verifica a instalação
agentic-squad --help
Node.js ≥ 20 requerido
npm ≥ 9
Acesso ao Jira (Admin)
GitHub App configurado
Passo 2

Executar o wizard de configuração

Na raiz do novo projeto (ou em qualquer diretório), execute:

bash 
agentic-squad init
Wizard

Os 7 passos do init

1
Configuração do Jira
Coleta e valida JIRA_BASE_URL, JIRA_EMAIL, JIRA_API_TOKEN e JIRA_PROJECT_KEY. Confirma conexão com o projeto Jira antes de prosseguir.
2
Configuração do GitHub App
Coleta GITHUB_OWNER, GITHUB_REPO, GITHUB_APP_ID, GITHUB_APP_PRIVATE_KEY (aceita caminho para arquivo .pem), GITHUB_APP_INSTALLATION_ID e branch padrão.
3
Detecção automática de stack
Lê os arquivos do repositório alvo via API GitHub e detecta linguagem, build tool e framework de testes. Pré-preenche os comandos de CI com base na stack encontrada. Se não detectado, permite preenchimento manual.
4
URL do serviço
Define SERVICE_URL — onde a squad está hospedada. Verifica a saúde do serviço antes de prosseguir. Se o serviço não responder, os Steps 5–7 continuam com alertas.
5
Instalação do workflow de CI
Instala .github/workflows/agent-dev.yml diretamente no repositório alvo via API GitHub (cria commit). Confirma antes de executar. Pode ser pulado para instalação manual.
6
Registro do webhook no Jira
Registra o webhook em Jira Admin → System → WebHooks com filtro JQL project = "KEY" e evento jira:issue_updated. URL gerada: SERVICE_URL/webhooks/jira?secret=.... Expira em 30 dias.
7
Smoke test end-to-end (opcional)
Cria uma história de teste no Jira, dispara o webhook e valida a resposta { queued: true }. Remove a história ao final. Confirma que o pipeline está funcionando de ponta a ponta.
✅
Ao final do wizard, todas as variáveis são salvas em .env.init para revisão. Copie-as para o .env do serviço e refaça o deploy.
Referência rápida

O que cada step faz

Step Ação Resultado
1 Configuração Jira Validação de acesso ao projeto
2 GitHub App Validação de permissões no repo
3 Detecção de stack CI_TEST_COMMAND e CI_COVERAGE_COMMAND configurados
4 URL do serviço SERVICE_URL salvo, saúde verificada
5 Workflow CI agent-dev.yml commitado no repo alvo
6 Webhook Jira Webhook registrado — squad escuta eventos
7 Smoke test Pipeline validado de ponta a ponta
