Processamento Paralelo de Histórias

Squad Agêntica — agentic-squad-heavy · Fase 2 · 2026-05-25

✅
Sim — o sistema suporta múltiplas histórias em paralelo, por design.

Duas ou mais histórias em statuses distintos (ou no mesmo status) são processadas de forma completamente independente. Não há estado compartilhado, locks globais ou dependências entre histórias diferentes.

⚙ Como funciona a independência entre histórias

Mecanismo 1
Webhook — caminho principal

Cada evento Jira dispara um job independente no orchestrator queue com jobId único por história e status. Histórias diferentes não compartilham jobs nem contexto de execução.

Mecanismo 2
BullMQ com concorrência configurada

Cada worker aceita múltiplos jobs simultâneos. Jobs de histórias diferentes executam em paralelo dentro do mesmo processo Node.js, sem bloqueio entre si.

Mecanismo 3
Reconciler divergence-driven

O reconciler (90s) só re-dispara jobs quando detecta que o Jira avançou além do banco. Enquanto um agente está rodando, Jira e DB estão no mesmo status — o reconciler passa direto sem duplicar trabalho.

Mecanismo 4
Isolamento completo por história

Cada história possui seus próprios registros em stories, agent_runs e artifacts. Não há tabelas ou variáveis globais compartilhadas entre histórias distintas.

🔄 Por que o Reconciler não duplica trabalho

Lógica de divergência — trecho de reconciler.ts:84
1
Busca no banco todas as histórias com status em andamento
2
Consulta o estado real de cada história no Jira via API
3
if (jiraCurrentStatus === story.jiraStatus) continue; — se Jira == banco, não faz nada (agente em execução, tudo certo)
4
Só reenfileira se o Jira estiver à frente do banco (jiraOrder > dbOrder) — sinal de webhook perdido
// reconciler.ts — verificação de divergência por história for (const story of dbStories) { const jiraCurrentStatus = jiraIssue.fields.status.name; // ← agente em execução: Jira == banco → não faz nada if (jiraCurrentStatus === story.jiraStatus) continue; if (jiraOrder > dbOrder) { // webhook perdido → reenfileira com jobId deduplicado await orchestratorQueue.add('jira:transition', { ... }, { jobId: `reconcile-${story.jiraKey}-${JIRA_TO_DB_STATUS[jiraCurrentStatus]}`, }); } }

⚠ Limites práticos a considerar

Limite Valor atual Impacto Status
Concorrência PO 3 histórias simultâneas A 4ª história em "Em Refinamento" entra na fila BullMQ e aguarda um slot liberar OK para testes
Concorrência LT 3 histórias simultâneas Idem para o Agente LT OK para testes
Rate limit Anthropic ~50 req/min (Tier 1) Muitas histórias simultâneas consumindo Claude API podem sofrer throttling — a fila BullMQ absorve picos naturalmente via retentativas com backoff exponencial Monitorar
Render free tier 1 instância Não há escalonamento horizontal — os 3 slots de cada worker rodam no mesmo processo; suficiente para o volume atual de desenvolvimento Planejado

Como escalar quando necessário

Se o squad crescer para 10+ histórias simultâneas, o único ajuste necessário é aumentar os valores de concurrency nos workers:

// src/agents/po.ts — ajuste de concorrência new Worker('agent-po', processPoJob, { connection: redisConnection, concurrency: 10, // era 3 — aumentar conforme demanda }); // src/index.ts — orquestrador new Worker('orchestrator', processJob, { connection: redisConnection, concurrency: 20, // era 5 });

O Redis (Upstash) e o PostgreSQL (Supabase) não precisam de nenhuma mudança — já suportam múltiplas conexões concorrentes. O gargalo para cargas maiores seria o plano do Render (uma instância), resolvido com upgrade para um plano com múltiplas instâncias e balanceamento de carga via Redis como broker compartilhado.
